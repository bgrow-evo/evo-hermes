# Studio profile

A Hermes **profile** (a separate `HERMES_HOME`) that runs the evo Photo Workflow
on a daily schedule. It runs as its own s6-supervised gateway **inside the same
`hermes` container** — separate profile, no separate container.

## What it does

A daily cron job (06:00 by default) runs the `studio-daily-pipeline` skill, which
attempts the full photo pipeline from the
[distilled playbook](../../../photo-workflow-cowork-playbook-distilled):

1. Daily Report Refresh → 2. Vendor Image Sourcing → 3. Image Editing & Processing
→ 4. **Package** a PIM-ready ZIP (it does **not** upload to PIM).

Human-only gates (VPN file import, OAuth, image-order confirmation, PIM upload) are
recorded as blockers, not faked. Final deliverable each day:

```
/opt/data/outbox/studio/<YYYY-MM-DD>/<Brand>_pim-ready.zip   (host: ~/.hermes/outbox/studio/...)
/opt/data/outbox/studio/<YYYY-MM-DD>/MANIFEST.md
```

## Layout (this scaffold — version controlled)

```
profiles/studio/
├── README.md              # this file
├── SOUL.md                # persona / mission
├── .env                   # vendor DAM credentials
├── config.overrides       # key/value config applied at deploy time
└── skills/
    ├── studio-daily-pipeline/   # orchestration skill (cron entrypoint)
    └── evo-image-processing/    # Pillow batch processing + ZIP packaging
        └── scripts/{process_images.py, package_zip.py}
```

Deployed (in the volume, created by `install.ps1`):

```
~/.hermes/profiles/studio/
├── config.yaml            # generated by `profile create`, then overridden
├── SOUL.md .env skills/   # copied from this scaffold
├── playbook/              # copied from photo-workflow-cowork-playbook-distilled
├── cron/jobs.json         # the daily job
└── state.db sessions/ ... # per-profile runtime state
```

## Lifecycle

`install.ps1` deploys this profile on every run (idempotent):

1. `docker exec hermes hermes profile create studio` (skipped if it exists) —
   registers the `gateway-studio` s6 slot.
2. Copies this scaffold + the distilled playbook into `~/.hermes/profiles/studio/`.
3. Applies `config.overrides` via `studio config set`.
4. Ensures the daily cron job exists.
5. Starts the studio gateway (`hermes -p studio gateway start`) and marks it
   running so the container's boot reconciler auto-restores it after rebuilds.

Because `~/.hermes` is the bind-mounted volume, the profile and its state persist
across `docker compose build` / image rebuilds — the apt image tools come from the
rebuilt image, the profile/playbook/cron from the volume.

## Operate

```powershell
# List gateways (default + studio) and status
docker exec hermes hermes gateway list

# Tail studio gateway logs
docker compose exec hermes bash -c "tail -f /opt/data/logs/gateways/studio/current"

# Cron: list / run-now / pause
docker exec hermes hermes -p studio cron list
docker exec hermes hermes -p studio cron run <job_id>     # fire on next tick
docker exec hermes hermes -p studio cron pause <job_id>

# Grab today's output
ls ~/.hermes/outbox/studio/
```

## Delivery note (Teams)

The cron delivers its run summary to `local` (written under
`~/.hermes/profiles/studio/cron/output/`), and the ZIP lands in the outbox above.
The studio gateway does **not** bind the Teams webhook port (3978 is owned by the
default profile's bot), so it does not post to Teams directly. To push the daily
summary/ZIP into a Teams channel, either pick it up from the outbox, or wire a
Power Automate flow watching `outbox/studio/` (the playbook already recommends
Power Automate for the Teams-posting step). Update `--deliver` on the cron job if a
reachable target is configured later.
