## Current Blocker: s6-log crash loop → gateways won't stay up

### Root cause
The Hermes container uses **s6-overlay** as its init system. Each gateway service
(`gateway-default`, `gateway-studio`) has an associated `s6-log` logger that writes
to `/opt/data/logs/gateways/<profile>/`.

`/opt/data` is mounted from **Azure Files SMB**, which does **not support POSIX
`fchmod`**. When `s6-log` tries to rotate the log file it calls `fchmod()` on the
`current` file → `EPERM` → s6-log crashes → the pipe to the gateway process breaks
→ gateway receives `SIGPIPE` and dies → s6-supervise restarts it → loop.

ACA log evidence (repeating continuously):
```
s6-log: fatal: unable to fd_chmod /opt/data/logs/gateways/studio/current: Operation not permitted
s6-log: fatal: unable to fd_chmod /opt/data/logs/gateways/default/current: Operation not permitted
WARNING gateway.run: Shutdown context: signal=SIGTERM ... parent_cmdline='s6-supervise gateway-studio'
```

The gateways themselves DO start (confirmed by `s6-supervise gateway-studio` in log),
but die within seconds due to the broken log pipe.

## What We Tried

### 1. Runtime patch in `hermes-aca-restore.sh` (FAILED — scripts not found)
Added a loop to patch `s6` gateway log run scripts at container startup:
```bash
for _svc in /etc/services.d/*/log/run; do
  grep -qE 'logs/gateways|HERMES_HOME' "$_svc" && patch to /tmp/hermes-logs/...
done
```
**Result**: Loop runs but finds NO files. The base image (`nousresearch/hermes-agent:latest`)
does NOT put log scripts in `/etc/services.d/*/log/run`. Their location is unknown.
The restore script confirms "Patching..." is logged but no `[before]`/`Patched` lines follow.

### 2. Build-time `sed` patch in `Dockerfile` (REMOVED — ineffective)
Tried statically patching log scripts during image build. Removed because the scripts
use `$HERMES_HOME` (env var) so `sed` on literal path found nothing.

### 3. EmptyDir volume via ACA REST PATCH (FAILED — silently ignored)
The correct fix is to mount an **ACA EmptyDir** (ephemeral tmpfs) volume at
`/opt/data/logs/gateways/`. This overlays the Azure Files mount with local tmpfs
at that exact subpath, so `fchmod` succeeds.

`Deploy-HermesAzure.ps1` was updated to include the EmptyDir in both `-Setup` and
`-Push` PATCH bodies:
```json
{
  "properties": {
    "template": {
      "volumes": [
        {"name": "hermes-data",   "storageType": "AzureFile", "storageName": "hermes-data"},
        {"name": "gateway-logs",  "storageType": "EmptyDir"}
      ],
      "containers": [{
        "name": "hermes",
        "volumeMounts": [
          {"volumeName": "hermes-data",   "mountPath": "/opt/data"},
          {"volumeName": "gateway-logs",  "mountPath": "/opt/data/logs/gateways"}
        ]
      }]
    }
  }
}
```

**Result**: `az rest --method PATCH` returns exit 0 but the volume is **never applied**.
`az containerapp show --query "properties.template.volumes"` always returns only
`hermes-data`. Also tried `storageType: "Ephemeral"` — same result. Also tried
`az rest --method PUT` with full spec — fails with:
```
ContainerAppSecretInvalid: secret 'acrhermessbxazurecrio-acrhermessbx' invalid:
value or keyVaultUrl and identity should be provided
```
(The GET response redacts the ACR password secret value, so PUT with it fails validation.)

## What Needs to Be Done

### Option A — Apply EmptyDir volume (RECOMMENDED)
Need to mount an ephemeral (tmpfs) volume at `/opt/data/logs/gateways/` inside the
ACA container. This overlays the Azure Files path with local storage that supports `fchmod`.

The `az rest PATCH` approach silently fails. Alternatives to investigate:
1. **Azure Portal** — Containers → Volumes tab → add emptyDir volume and mount manually
2. **Bicep/ARM template** — redeploy with explicit `emptyDir` volume in the ARM spec
3. **`az containerapp update --yaml`** — provide a YAML file with the full template
   (challenge: registry password secret is redacted in GET, need to re-supply it)

The correct ARM JSON for an emptyDir volume is:
```json
{ "name": "gateway-logs", "storageType": "EmptyDir" }
```
with container volumeMount:
```json
{ "volumeName": "gateway-logs", "mountPath": "/opt/data/logs/gateways" }
```

### Option B — Find and patch the s6-log invocation scripts
If EmptyDir is hard to add, an alternative is to patch the s6 gateway log run scripts
inside the container image to write to `/tmp/hermes-logs/` (local tmpfs) instead of
`/opt/data/logs/gateways/`.

**The scripts are NOT in** `/etc/services.d/*/log/run`. They are somewhere in the
base image (`nousresearch/hermes-agent:latest`). Need to find where.

To investigate, exec into a running container:
```bash
az containerapp exec --name aca-hermes --resource-group rg-hermes-sbx \
  --command "find /etc -name 'run' 2>/dev/null | xargs grep -l 'gateways' 2>/dev/null"
```
OR pull the base image locally and inspect:
```bash
docker pull nousresearch/hermes-agent:latest
docker run --rm --entrypoint sh nousresearch/hermes-agent:latest \
  -c "find /etc -name 'run' | xargs grep -l 'gateways' 2>/dev/null"
```
Once found, patch the scripts in the `Dockerfile` with `sed` or `RUN` commands.

### Option C — Disable gateway logging entirely
Add a `HERMES_LOG_DIR=/tmp/hermes-logs` (or equivalent) environment variable if
Hermes supports it. Or configure the gateway s6 log service to write to `/dev/null`.
